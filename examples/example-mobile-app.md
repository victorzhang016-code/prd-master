# 用户邀请奖励功能

**版本** v1.0　**日期** 2025-04-11　**状态** 待评审

---

## 一、为什么做这个

**现在的问题**：目前 70% 的新用户来自自然流量，获客成本持续上升（CPM 从 ¥45 涨到 ¥78），而老用户推荐转化率仅为 3%，远低于行业平均 12%。

**我们要做什么**：上线双向邀请奖励机制，邀请者和被邀请者各获得 7 天会员。

**做完之后**：用户会感受到「推荐朋友有实实在在的好处」，而不是现在的「感谢你的推荐」空洞文案。

---

## 二、怎么做

### 核心流程

1. 用户点击「邀请好友」→ 生成专属邀请码/链接
2. 好友通过链接下载 App → 注册时自动填入邀请码
3. 好友完成首单 → 双方各获得 7 天会员

### 关键细节

**邀请码设计**
- 格式：6 位字母数字混合（如 XP7K9M）
- 过期：邀请码永久有效，但奖励仅限首次绑定

**奖励发放时机**
- 好友完成首单（支付成功）后即时发放
- 通过 Push + 站内信通知双方

**防刷机制**
- 同一设备/IP 注册视为同一用户
- 邀请关系建立后不可更改

## 三、技术方案

| 方案 | 好处 | 坏处 | 结论 |
|------|------|------|------|
| 邀请码填写 | 实现简单 | 用户可能忘记填 | ❌ |
| 链接带参数自动绑定 | 体验流畅 | 需要处理 Universal Link | ✅ |
| 二维码扫描 | 适合线下 | 线上场景麻烦 | ❌ 作为备选 |

**我们选方案 2**，因为体验最关键，技术成本可接受。

## 四、字段定义

| 字段 | 类型 | 说明 |
|------|------|------|
| `inviter_id` | string | 邀请人用户ID |
| `invitee_id` | string | 被邀请人用户ID |
| `invite_code` | string(6) | 6位邀请码 |
| `invite_link` | string | 带追踪参数的链接 |
| `status` | enum | pending/converted/expired |
| `reward_sent_at` | timestamp | 奖励发放时间 |

---

## 五、边界情况

| 如果…… | 处理方式 |
|--------|---------|
| 用户 A 邀请 B，B 又邀请 C | 只有 A-B 建立关系，B-C 是另一个独立关系 |
| 用户点击多个邀请链接 | 以最后一次点击为准，之前的关系自动解除 |
| 好友注册但未下单 | 关系保留 30 天，超时自动标记为 expired |
| 用户卸载重装 | 邀请关系不变，存储在服务端 |

## 七、怎么算做好了

**北极星指标**：推荐带来的新用户占比

| 指标 | 基准值 | 目标值 | 观测窗口 |
|------|--------|--------|---------|
| 推荐新用户占比 | 3% | 10% | 上线后 30 天 |
| 邀请码生成率 | - | 20%（日活用户）| 上线后 7 天 |
| 邀请完成率 | - | 15%（生成到注册）| 上线后 14 天 |

**过程指标**：

| 过程指标 | 目标 | 说明 |
|---------|------|------|
| 邀请页 PV | 日活 20% 点击 | 说明入口可见性够 |
| 链接打开率 | 30% | 说明文案吸引力 |
| 注册转化率 | 40% | 说明落地页质量 |

**验收标准**：
- [ ] 邀请功能上线后 30 天内，推荐新用户占比达到 10%
- [ ] 无重大客诉（定义为涉及邀请奖励的客服工单 < 5%）
- [ ] 系统无邀请关系错乱 bug

## 八、埋点

| 事件名 | 触发时机 | 必传字段 | 用来验证 |
